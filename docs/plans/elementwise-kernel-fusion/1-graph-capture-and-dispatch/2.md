# Part 1.2 — Lazy machinery + dispatch (the inductor seam)

> Builds on the refactors in `1.md`. This part introduces actual
> deferral: the `LazyFunction` graph (§0), the abstract
> `RealizationBackend` seam in `tensor`, the concrete inductor backend,
> the `TensorImpl::lazy_` field, the op-shim `defer` branch, and the
> `data()` realize chokepoint.
>
> Option A (decided): **capture every op on CUDA when a backend is
> registered**, regardless of fusibility. The fusible/non-fusible split is
> Part 2's job (merge + partition). Non-fusible ops record a boundary node
> and realize via a `run_eager` bridge to their existing `*_stub()`.
>
> Realize itself (DFS partition, codegen, NVRTC) is a **stub** here — this
> part proves the *record + trigger + eager-bridge* loop end to end. A
> recorded graph, when forced, must reproduce the eager result by walking
> operands and calling `run_eager`. Real fusion lands in Part 2.

## Depends on `1.md`

- `data()` is non-`const`/non-`noexcept`; `realize()`/`sync()` stubs exist.
- `meta_handler` can `infer()` metadata without allocating.
- `Storage` tolerates 0 bytes; `TensorImpl` ctor bypasses the size check
  for a 0-byte buffer.
- Shape ops route through `ops::`.
- Captured operands are held by `shared_ptr<TensorImpl>`.

## 0. Decided graph model

> The canonical graph design; §1–§7 detail the mechanism consistent with
> it. No `LazyNode`, no `OpKind`, `LazyFunction` in `tensor`, metadata on
> `TensorImpl`.

**Lens — shadow the autograd graph exactly.** `TensorImpl` ↔ `VariableImpl`
(the vertex, carries data + metadata), `LazyFunction` ↔ `Function` (the op,
carries inputs + how to compute), per-op `LazyFunction` subclass ↔
`ClampBackward` (op params as fields). Autograd has no separate node class;
neither do we.

1. **No separate `LazyNode`.** `TensorImpl` *is* the vertex; `is_lazy()`
   (`lazy_ != nullptr`) separates a realized leaf (has `data_`, terminates
   DFS) from an op node (0-byte storage, real metadata). DFS stops on
   `!is_lazy()`, not on a leaf-wrapper object.

2. **Metadata lives on `TensorImpl`, never duplicated in `LazyFunction`.**
   See `1.md §2` — each op's `infer_output()` builds the 0-byte output impl
   carrying shape/dtype/device; operands are `TensorImpl`s carrying their
   own. So realize/codegen reach every node's metadata without the function
   storing any.

3. **No `OpKind` enum.** Op behavior (`run_eager`, `codegen_expr`) is a
   vtable on the subclass (like autograd's `Function`s) — no execution
   dispatch on a kind tag. The only tag is a coarse, stable `is_fusible()`
   / `OpCategory` label (~5 values, not one-per-op) for Part 2's
   partitioner.

4. **`LazyFunction` + all op subclasses live in `tensor`, not inductor.**
   Forced by direct construction `record(make_shared<AddFn>(...))`: the
   op-shim `ops::add` is public `tensor` API and can only name `AddFn` if
   `AddFn` is a `tensor` type. The inductor-side alternatives are the two
   we rejected — a type-erased key (the deleted `OpKind`) or a per-op
   virtual on the backend (~30, cross-module surface blow-up).
   - **Vocabulary vs machinery.** `tensor` holds op *vocabulary* (per-op,
     grows with ops: `inputs`, `run_eager`→`*_stub()`, `codegen_expr`
     string, `is_fusible` label). `inductor` holds the fusion *machinery*
     (fixed: partition, NVRTC codegen, module cache, launch). Adding an op
     = one file, one module. Cross-module surface = exactly one seam,
     `RealizationBackend`.

5. **`record` returns `shared_ptr<TensorImpl>`, is a free fn, no backend.**
   Each subclass owns its shape via `infer_output()` (`1.md §2`), so
   `record` just calls it, sets `impl->lazy_ = fn`, returns the impl. The
   singleton `RealizationBackend* backend()` is still needed — for
   `realize()` (NVRTC) and the defer gate (`backend() && device==CUDA`) —
   just not for construction.

### Sketch

```cpp
// tensor module
struct LazyFunction {                          // abstract
  std::vector<std::shared_ptr<TensorImpl>> inputs;
  virtual ~LazyFunction() = default;
  virtual std::shared_ptr<TensorImpl> infer_output() const = 0; // 0-byte impl
  virtual void run_eager(TensorImpl& out) = 0;                  // → *_stub()
  virtual std::string codegen_expr() const = 0;                 // Part 2 seam
  virtual bool is_fusible() const = 0;                          // coarse label
};
struct ElementwiseBinaryFn : LazyFunction {    // broadcast infer once, here
  std::shared_ptr<TensorImpl> infer_output() const override;
  bool is_fusible() const override { return true; }
};
struct AddFn : ElementwiseBinaryFn {           // the whole op, one place
  void run_eager(TensorImpl& out) override;    // add_stub on realized inputs
  std::string codegen_expr() const override { return "{0}+{1}"; }
};

std::shared_ptr<TensorImpl> record(std::shared_ptr<LazyFunction> fn) {
  auto out = fn->infer_output();
  out->lazy_ = std::move(fn);
  return out;                                   // free fn in tensor, no backend
}

Tensor ops::add(const Tensor& a, const Tensor& b) {
  bool defer = backend() && a.device() == DeviceType::CUDA;
  if (!defer) return primitives::add(a, b);     // today's eager path
  return Tensor{ record(std::make_shared<AddFn>(
      std::vector{a.impl(), b.impl()})) };
}
```

### Pointers vs values

| Thing                          | Type                              | Why |
|--------------------------------|-----------------------------------|-----|
| `TensorImpl::lazy_`            | `shared_ptr<LazyFunction>`        | DAG, not tree (≈ `_grad_fn`) |
| `LazyFunction::inputs`         | `vector<shared_ptr<TensorImpl>>`  | keep producers alive to realize |
| shape / dtype / strides        | by value on `TensorImpl`          | already are |
| op params (clamp bounds, axis) | by value on the subclass          | ≈ `ClampBackward::min_val_` |

Edges point output→inputs only (acyclic) → plain `shared_ptr` everywhere,
no `weak_ptr` needed for the lazy graph.

### Open / deferred
- If Part 2 codegen needs `inductor`-private per-op state (beyond a format
  string), reintroduce it as a **visitor** (`node->emit(backend&)`) — still
  a virtual, still no enum, subclasses stay in `tensor`.
- "Realize an *earlier* node after it's been realized-and-cleared" is
  assumed never to happen — assert if violated (`index.md` open Q).

## 1. Abstract seam in `tensor` (no inductor include)

`Tensor`/`TensorImpl` must never include inductor. The **only** abstract
seam is `RealizationBackend`; the inductor lib provides its one concrete
subclass. `LazyFunction` and its op subclasses are `tensor` types (§0), so
they need no cross-module type-erasure — `record` is a free fn in `tensor`,
not a backend method.

```cpp
// tensor/realization_backend.hpp — declared in tensor module
namespace lmp::tensor {

class LazyFunction;    // concrete base + op subclasses live in tensor (§0)

struct RealizationBackend {                  // abstract; inductor subclasses
  virtual ~RealizationBackend() = default;
  virtual void realize(TensorImpl*) = 0;     // record is a free fn (§0), not here
};

RealizationBackend* backend();               // null unless registered
void register_backend(RealizationBackend*);  // inductor calls at startup
}
```

- The leaf/op-node split is gone (§0): `TensorImpl` is the vertex,
  `is_lazy()` distinguishes a realized leaf from an op node holding a
  `shared_ptr<LazyFunction>`.

**Files.** new `csrc/include/lamppp/tensor/realization_backend.hpp` and
`lazy_function.hpp`, wired into the tensor module.

## 2. `TensorImpl::lazy_` field

Add the deferred-graph pointer next to `Storage` (the field sketched in
`index.md`):

```cpp
struct TensorImpl {
  Storage data_;                       // realized bytes (0-byte while lazy)
  std::shared_ptr<LazyFunction> lazy_; // the pending op; null on eager path
  bool is_lazy() const { return lazy_ != nullptr; }
};
```

- Invariant: exactly one of `{data_ populated, lazy_ set}`. A lazy impl
  carries the 0-byte on-device `Storage` from `1.md §3`.
- Eager path never touches `lazy_` → one null-check on the hot path.
- Decide union-vs-both-members (the `index.md` TODO): keep both members
  (simpler; `Storage` already cheap when 0-byte) unless profiling says
  otherwise.

**Files.** `csrc/include/lamppp/tensor/tensor_impl.hpp`,
`csrc/src/tensor/tensor_impl.cpp`.

## 3. `LazyFunction` (tensor lib)

In `tensor` (§0), not inductor — analogue of the autograd `Function`, but
for fusion: describes *what computation is pending*, not differentiation.
See the §0 sketch for the full shape.

- `LazyFunction` (abstract base): carries `inputs`
  (`vector<shared_ptr<TensorImpl>>`) plus op-specific params (e.g. clamp
  bounds), and the virtuals `infer_output()`, `run_eager()`,
  `codegen_expr()`, `is_fusible()`. No `OpKind`, no output metadata — that
  lives on the `TensorImpl` the op produces (§0 #2).
- One concrete subclass per op (shape inference shared on a category base
  like `ElementwiseBinaryFn`). `run_eager` bridges to the matching
  `*_stub()` on realized operands — the boundary/realize path for Part 1b
  and for non-fusible ops forever.

**Files.** new in the tensor module:
`csrc/include/lamppp/tensor/lazy_function.hpp`,
`csrc/src/tensor/lazy_function.cpp` (+ per-op subclasses alongside the
existing `ops::`). The NVRTC backend stays in inductor
(`csrc/src/inductor/nvrtc/`).

## 4. `record` (free fn) + concrete backend `realize` (realize stubbed)

**record** (free fn in `tensor`, §0) — takes a `shared_ptr<LazyFunction>`,
calls its `infer_output()` (which computes output metadata *now* via
`meta_handler::infer()` — `type_upcast`, shape, broadcast flags — and
allocates a `TensorImpl` with a **0-byte** `Storage`), sets
`impl->lazy_ = fn`, and returns the lazy impl. No bytes, no backend.
Operands held as `shared_ptr<TensorImpl>` (`1.md §6`) so producers stay
alive.

`NVRTCInductorBackend : RealizationBackend`, registered into `tensor` at
startup, provides **realize** only.

**realize(impl)** — **Part 1b stub form:** DFS the operand graph; for each
operand that is itself lazy, realize it first; then call the node's
`run_eager` bridge (→ existing `*_stub()`), write the resulting `Storage`
back into the impl, and clear `lazy_`. This reproduces the eager answer
through the recorded graph with **no fusion**. Part 2 replaces the body
with the fusion-group partition + NVRTC codegen + single-kernel launch,
keeping `run_eager` as the boundary path.

Open Q (record in `index.md`): assume we never `data()` an *earlier*
node after it's been realized-and-cleared — hardcode that assumption,
assert if violated.

**Files.** `csrc/src/inductor/nvrtc/backend.cpp` (record + realize),
backend singleton question — make `backend()` return a single registered
instance.

## 5. Op shim: the two-arm `defer` branch

Every op gateway (`ops::add`, `ops::neg`, …, and the shape ops from
`1.md §1`) gets the Option-A branch:

```cpp
Tensor ops::add(const Tensor& a, const Tensor& b) {
  bool defer = backend() && a.device() == DeviceType::CUDA;  // device-only gate
  if (!defer) return primitives::add(a, b);                  // today's eager path
  return Tensor{ record(std::make_shared<AddFn>(
      std::vector{a.impl(), b.impl()})) };
}
```

- CPU and no-backend cases always take the eager arm → unchanged. Only
  CUDA-with-backend records.
- Applies to **every** op (Option A), not just elementwise — boundary ops
  record too and realize via `run_eager`.
- `a.impl()` here is the `shared_ptr<TensorImpl>` from
  `UnsafeTensorAccessor::getImpl` (`1.md §6`).

**Files.** `csrc/src/tensor/native/expand_ops.cpp` (binary),
`unary_ops.cpp`, `shape_ops.cpp`, `reduct_ops.cpp`, `matrix_ops.cpp`,
`conv_ops.cpp` — each `ops::*` definition gains the branch. Consider a
small helper/macro to avoid hand-repeating the two-arm shim across ~30
ops.

## 6. `data()` realize chokepoint

Fill the `1.md §5` stub: `data()` forces realization, then returns bytes.

```cpp
void* TensorImpl::data() {
  if (is_lazy()) backend()->realize(this);   // force, then fall through
  return data_.data();
}
```

- All host-side raw-byte access (`to_vector`, `.index()`, `print`,
  autograd reading saved tensors) bottoms out in `data()`, so this one
  chokepoint catches the whole host-read class.
- `realize()`/`sync()` public methods (from `1.md §5`) now delegate to
  `backend()->realize(this)`.
- Non-`data()` consumers (non-elementwise kernels that read operands
  directly, backward) still need their own triggers — flagged, handled as
  they surface / in Part 2's boundary realize.

**Files.** `csrc/src/tensor/tensor_impl.cpp`, `tensor.cpp`.

## 7. Static-registration / linker hazard

Already bitten once (commit `1cd2975` "linker error fix"). If inductor is
a separate shared lib and nothing in `tensor` references its symbols, the
linker discards the `register_backend()` static-init TU — `backend()`
stays null and every op silently takes the eager arm.

Plan now (don't discover it at link time):
- Use `--whole-archive` / `-u` on the inductor lib, **or**
- Expose an explicit `lmp_inductor_init()` that `tensor` (or the app)
  calls, making the registrar transitively referenced.
- The cross-`.so` boundary weakens the usual static-registration macro
  trick (`dispatch_stub.hpp` comment) — prefer the explicit-init route.

**Files.** `CMakeLists.txt` (link flags / new inductor target), inductor
backend registration TU.

## Verification (end-to-end, no fusion yet)

1. No backend registered → identical to today (eager everywhere).
2. Backend registered, CUDA tensors → ops return lazy impls
   (`is_lazy()==true`, 0-byte storage), and a forcing read (`to_vector`,
   `print`, `.index()`) yields **bit-identical results to the eager
   path**, proving record + `run_eager` realize is correct before any
   codegen exists.
3. CPU tensors with backend registered → still eager (defer gate is
   device-keyed).

When this is green, Part 2 swaps the `realize` body from "walk + run_eager
per node" to "partition into fusion groups + one NVRTC kernel per group",
with `run_eager` retained as the boundary path.
