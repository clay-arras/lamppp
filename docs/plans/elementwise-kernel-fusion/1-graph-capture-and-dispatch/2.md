# Part 1.2 — Lazy machinery + dispatch (the inductor seam)

> Builds on the refactors in `1.md`. This part introduces actual
> deferral: the `LazyFunction` graph (§0), the abstract
> `LazyBackend` seam in `tensor`, the concrete inductor backend,
> the `TensorImpl::lazy_` field, the op-shim `defer` branch, and the
> `data()` realize chokepoint.
>
> Option A (decided): **capture every op when a lazy backend is registered
> for that op's device**, regardless of fusibility — the gate is
> `backend(device) != nullptr`, no hardcoded device. The fusible/non-fusible
> split is Part 2's job (merge + partition). Non-fusible ops record a
> boundary node and realize via a `run_eager` bridge to their existing
> `*_stub()`. **Carve-out:** cross-device `.to` is *not* captured — it stays
> eager as the device barrier (see Decisions).
>
> Realize itself (DFS partition, codegen, NVRTC) is a **stub** here — this
> part proves the *record + trigger + eager-bridge* loop end to end. A
> recorded graph, when forced, must reproduce the eager result by walking
> operands and calling `run_eager`. Real fusion lands in Part 2.

## Decisions locked (2026-06-27)

> These supersede any conflicting sketch below; §0/§1/§5/§6/§7 text has been
> updated to match.

1. **Per-device backend registry, not a single global `backend()`.** Mirror
   the existing `DispatchStub` (`dispatch_stub.hpp`): a
   `std::array<LazyBackend*, DeviceType::Count>` behind a Meyers-singleton
   accessor. `backend(DeviceType)` returns the backend for that device (or
   `nullptr`); `register_backend(DeviceType, LazyBackend*)` fills a slot.
   Non-owning raw pointers (backends are static instances in their own TU,
   exactly like the kernel fn pointers). A **separate** registry — do not
   overload `DispatchStub` (it holds free-fn pointers; this holds an object
   pointer).
2. **The defer gate is `backend(a.device()) != nullptr` — no `device ==
   CUDA` literal anywhere.** "Should this op defer?" = "is a lazy backend
   registered for this tensor's device?" CUDA is just the first backend to
   register; a CPU lazy backend can register on the same seam (and makes the
   whole record→realize loop exercisable without a GPU).
3. **`realize` is per-device.** Forcing dispatches through the node's own
   device: `backend(impl->device())->realize(impl)`. One device per node →
   one backend per node; no node is ever owned by two.
4. **Cross-device `.to` stays eager (carve-out to Option A).** Both
   byte-moving cross-device paths — `ops::to` (`memory_ops.cpp`) and
   `TensorImpl::copy` (`tensor_impl.cpp:106`, used by `variable.cpp:45` /
   `tensor.cpp:47`) — already bottom out in `data()`. So a deferred graph
   **never spans a device boundary**: `.to` forces its input subgraph (on
   that subgraph's device, via the §6 chokepoint) and emits a *realized*
   leaf on the target device. Every lazy graph is single-device by
   construction; exactly one backend ever owns it. Therefore **do not
   capture `to` as a `LazyFunction`** — leaving it eager makes the invariant
   explicit (no DFS edge ever crosses a device).
5. **No defer macro/helper.** With the gate reduced to one line
   (`if (backend(a.device())) ...`), each op shim just writes it inline — no
   `LMP_MAYBE_DEFER`, no helper fn.
6. **§7 (cross-`.so` static-registration / linker) is deferred — out of
   scope for this pass.** Use the existing static-registrar style
   (`LMP_REGISTER_DISPATCH`-shaped) for now; robust cross-`.so` survival is
   scoped separately later.
7. **Real accessors.** Capture uses
   `detail::UnsafeTensorAccessor::getImpl(t)` (→ `shared_ptr<TensorImpl>`)
   and `::fromImpl(ptr)` (→ `Tensor`) — not the `a.impl()` shorthand in the
   older sketches.

## Depends on `1.md`

- `data()` is non-`const`/non-`noexcept`; `realize()`/`sync()` stubs exist.
- `meta_handler` can `infer()` metadata without allocating.
- `Storage` tolerates 0 bytes; `TensorImpl` ctor bypasses the size check
  for a 0-byte buffer.
- Shape ops route through `ops::`.
- Captured operands are held by `shared_ptr<TensorImpl>`.

## 0. Decided graph model

> The canonical graph design; §1–§7 detail the mechanism consistent with
> it. No `LazyNode`, no `OpKind`, `LazyFunction` in `tensor`, metadata on
> `TensorImpl`.

**Lens — shadow the autograd graph exactly.** `TensorImpl` ↔ `VariableImpl`
(the vertex, carries data + metadata), `LazyFunction` ↔ `Function` (the op,
carries inputs + how to compute), per-op `LazyFunction` subclass ↔
`ClampBackward` (op params as fields). Autograd has no separate node class;
neither do we.

1. **No separate `LazyNode`.** `TensorImpl` *is* the vertex; `is_lazy()`
   (`lazy_ != nullptr`) separates a realized leaf (has `data_`, terminates
   DFS) from an op node (0-byte storage, real metadata). DFS stops on
   `!is_lazy()`, not on a leaf-wrapper object.

2. **Metadata lives on `TensorImpl`, never duplicated in `LazyFunction`.**
   See `1.md §2` — each op's `infer_output()` builds the 0-byte output impl
   carrying shape/dtype/device; operands are `TensorImpl`s carrying their
   own. So realize/codegen reach every node's metadata without the function
   storing any.

3. **No `OpKind` enum.** Op behavior (`run_eager`, `codegen_expr`) is a
   vtable on the subclass (like autograd's `Function`s) — no execution
   dispatch on a kind tag. The only tag is a coarse, stable `is_fusible()`
   / `OpCategory` label (~5 values, not one-per-op) for Part 2's
   partitioner.

4. **`LazyFunction` + all op subclasses live in `tensor`, not inductor.**
   Forced by direct construction `record(make_shared<AddFn>(...))`: the
   op-shim `ops::add` is public `tensor` API and can only name `AddFn` if
   `AddFn` is a `tensor` type. The inductor-side alternatives are the two
   we rejected — a type-erased key (the deleted `OpKind`) or a per-op
   virtual on the backend (~30, cross-module surface blow-up).
   - **Vocabulary vs machinery.** `tensor` holds op *vocabulary* (per-op,
     grows with ops: `inputs`, `run_eager`→`*_stub()`, `codegen_expr`
     string, `is_fusible` label). `inductor` holds the fusion *machinery*
     (fixed: partition, NVRTC codegen, module cache, launch). Adding an op
     = one file, one module. Cross-module surface = exactly one seam,
     `LazyBackend`.

5. **`record` returns `shared_ptr<TensorImpl>`, is a free fn, no backend.**
   Each subclass owns its shape via `infer_output()` (`1.md §2`), so
   `record` just calls it, sets `impl->lazy_ = fn`, returns the impl. The
   per-device registry `LazyBackend* backend(DeviceType)` is still needed —
   for `realize()` (NVRTC) and the defer gate (`backend(device) != nullptr`)
   — just not for construction.

### Sketch

```cpp
// tensor module
struct LazyFunction {                          // abstract
  std::vector<std::shared_ptr<TensorImpl>> inputs;
  virtual ~LazyFunction() = default;
  virtual std::shared_ptr<TensorImpl> infer_output() const = 0; // 0-byte impl
  virtual void run_eager(TensorImpl& out) = 0;                  // → *_stub()
  virtual std::string codegen_expr() const = 0;                 // Part 2 seam
  virtual bool is_fusible() const = 0;                          // coarse label
};
struct ElementwiseBinaryFn : LazyFunction {    // broadcast infer once, here
  std::shared_ptr<TensorImpl> infer_output() const override;
  bool is_fusible() const override { return true; }
};
struct AddFn : ElementwiseBinaryFn {           // the whole op, one place
  void run_eager(TensorImpl& out) override;    // add_stub on realized inputs
  std::string codegen_expr() const override { return "{0}+{1}"; }
};

std::shared_ptr<TensorImpl> record(std::shared_ptr<LazyFunction> fn) {
  auto out = fn->infer_output();
  out->lazy_ = std::move(fn);
  return out;                                   // free fn in tensor, no backend
}

Tensor ops::add(const Tensor& a, const Tensor& b) {
  using detail::UnsafeTensorAccessor;
  LMP_CHECK(a.device() == b.device()) << "Tensors are on different devices";
  if (backend(a.device()))                      // a backend registered for this device?
    return UnsafeTensorAccessor::fromImpl(record(std::make_shared<AddFn>(
        std::vector<std::shared_ptr<TensorImpl>>{
            UnsafeTensorAccessor::getImpl(a), UnsafeTensorAccessor::getImpl(b)})));
  return /* today's eager add_stub() path, unchanged */;
}
```

### Pointers vs values

| Thing                          | Type                              | Why |
|--------------------------------|-----------------------------------|-----|
| `TensorImpl::lazy_`            | `shared_ptr<LazyFunction>`        | DAG, not tree (≈ `_grad_fn`) |
| `LazyFunction::inputs`         | `vector<shared_ptr<TensorImpl>>`  | keep producers alive to realize |
| shape / dtype / strides        | by value on `TensorImpl`          | already are |
| op params (clamp bounds, axis) | by value on the subclass          | ≈ `ClampBackward::min_val_` |

Edges point output→inputs only (acyclic) → plain `shared_ptr` everywhere,
no `weak_ptr` needed for the lazy graph.

### Open / deferred
- If Part 2 codegen needs `inductor`-private per-op state (beyond a format
  string), reintroduce it as a **visitor** (`node->emit(backend&)`) — still
  a virtual, still no enum, subclasses stay in `tensor`.
- "Realize an *earlier* node after it's been realized-and-cleared" is
  assumed never to happen — assert if violated (`index.md` open Q).

## 0b. Inductor module scaffold (build wiring)

> Net-new module. Nothing in `index.md`/§1–§7 builds until a compilable,
> linkable `inductor_core` target exists and is reachable from the
> aggregate `lamppp` target. This step is **pure scaffolding** — an empty
> backend that registers itself and does nothing — so it can land and be
> verified (links, loads, `backend(dev)` non-null) before any real lazy code.

**What.** Create the inductor module as a sibling of `tensor`/`autograd`/
`nets`, mirroring the existing `tensor_core` CMake shape:

- New dirs: `csrc/src/inductor/` (sources) and
  `csrc/include/lamppp/inductor/` (public headers). The NVRTC backend will
  live under `csrc/src/inductor/nvrtc/` (§4b).
- New `csrc/src/inductor/CMakeLists.txt` defining
  `add_library(inductor_core SHARED ...)`, linking
  `PUBLIC lamppp_compile_config tensor_core` (it depends on `TensorImpl` /
  `LazyBackend` from tensor) and, under `LMP_ENABLE_CUDA`,
  `CUDA::cudart` **and `CUDA::nvrtc`** (codegen needs NVRTC at runtime).
- Top `CMakeLists.txt`: `add_subdirectory(csrc/src/inductor)` after
  `tensor`, and add `inductor_core` to the aggregate
  `target_link_libraries(lamppp INTERFACE ...)` list.

**Why.** `tensor_core` is a SHARED lib; the backend is a separate SHARED
lib that depends on it and is injected back at startup (§1/§7). The module
boundary is the whole point of "inductor" being separate from "tensor"
(`index.md`): tensor holds op *vocabulary*, inductor holds fusion
*machinery*, with `LazyBackend` (§1) the single seam between them.

**Dependency direction (one-way).** `inductor_core → tensor_core` only.
`tensor_core` must **not** link `inductor_core` (it only sees the abstract
`LazyBackend` from §1). This is what keeps `Tensor`/`TensorImpl` free of any
inductor include (§1) and avoids a link cycle.

**Gating.** The NVRTC backend is CUDA-only, but the *seam* is not — the gate
is `backend(device) != nullptr` (Decision 2), so which devices defer is a
function of what's registered, not a compile flag. Build `inductor_core`
regardless of `LMP_ENABLE_CUDA`: under CUDA it registers the NVRTC backend
for `DeviceType::CUDA`; with CUDA off it still builds (no NVRTC) and can host
a CPU lazy backend used to exercise the record→realize loop without a GPU
(Verification). The `add_subdirectory` is therefore unconditional; only the
NVRTC sources/links are guarded on `LMP_ENABLE_CUDA`.

**Linker hazard is real here, not later.** Because nothing in
`tensor_core` references `inductor_core` symbols, the static-init
registrar can be dropped at link time — see §7. §0b creates the target;
§7 makes sure its registrar survives. Keep them in mind together.

**Done when.** A trivial `inductor_core` (one TU, a no-op
`LazyBackend` subclass + `register_backend` call) compiles, links into
`lamppp`, and at runtime `tensor::backend(dev)` returns non-null after the
inductor lib is loaded — proving the cross-`.so` registration path before
any lazy logic exists.

**Files.** `csrc/src/inductor/CMakeLists.txt` (new),
`csrc/src/inductor/` + `csrc/include/lamppp/inductor/` (new dirs), top
`CMakeLists.txt` (`add_subdirectory` + `lamppp` link list).

## 1. Abstract seam in `tensor` (no inductor include)

`Tensor`/`TensorImpl` must never include inductor. The **only** abstract
seam is `LazyBackend`; the inductor lib provides its one concrete
subclass. `LazyFunction` and its op subclasses are `tensor` types (§0), so
they need no cross-module type-erasure — `record` is a free fn in `tensor`,
not a backend method.

```cpp
// tensor/lazy/lazy_backend.hpp — declared in tensor module
namespace lmp::tensor {

class LazyFunction;    // concrete base + op subclasses live in tensor (§0)

struct LazyBackend {                  // abstract; inductor subclasses
  virtual ~LazyBackend() = default;
  virtual void realize(TensorImpl*) = 0;     // record is a free fn (§0), not here
};

// Per-device registry (mirrors DispatchStub's std::array<fn, DeviceType::Count>,
// but holds a LazyBackend* per slot). Meyers-singleton table internally.
LazyBackend* backend(DeviceType dev);                 // nullptr if none for dev
void register_backend(DeviceType dev, LazyBackend*);  // inductor calls at startup
}
```

- The leaf/op-node split is gone (§0): `TensorImpl` is the vertex,
  `is_lazy()` distinguishes a realized leaf from an op node holding a
  `shared_ptr<LazyFunction>`.
- **Separate registry, mirroring `DispatchStub`** (Decision 1): a
  `std::array<LazyBackend*, DeviceType::Count>` behind a Meyers-singleton
  accessor in `lazy_backend.cpp`. Do not overload `DispatchStub` — it keys
  free-fn pointers; this keys an object pointer per device.

**Files.** new `csrc/include/lamppp/tensor/lazy/lazy_backend.hpp`,
`lazy_backend.cpp` (the registry), and `lazy_function.hpp`, wired into the
tensor module.

## 3. `LazyFunction` (tensor lib)

In `tensor` (§0), not inductor — analogue of the autograd `Function`, but
for fusion: describes *what computation is pending*, not differentiation.
See the §0 sketch for the full shape.

- `LazyFunction` (abstract base): carries `inputs`
  (`vector<shared_ptr<TensorImpl>>`) plus op-specific params (e.g. clamp
  bounds), and the virtuals `infer_output()`, `run_eager()`,
  `codegen_expr()`, `is_fusible()`. No `OpKind`, no output metadata — that
  lives on the `TensorImpl` the op produces (§0 #2).
- One concrete subclass per op (shape inference shared on a category base
  like `ElementwiseBinaryFn`). `run_eager` bridges to the matching
  `*_stub()` on realized operands — the boundary/realize path for Part 1b
  and for non-fusible ops forever.

**Files.** new in the tensor module:
`csrc/include/lamppp/tensor/lazy_function.hpp`,
`csrc/src/tensor/lazy_function.cpp` (+ per-op subclasses alongside the
existing `ops::`). The NVRTC backend stays in inductor
(`csrc/src/inductor/nvrtc/`).

### 3a. Folder layout — mirror `autograd/functions/`

```
tensor/lazy/
  lazy_backend.hpp     (§1 seam)
  lazy_function.hpp    (abstract base ONLY)
  record.hpp/.cpp      (record() free fn; needs base only)
  functions/           elementwise_binary, elementwise_unary,
                       reduct, matrix, shape, conv   (base + per-op subclasses)
```
Category bases (`ElementwiseBinaryFn`, …) sit in their `functions/*` file.
Op-params are fields on the subclass (`ClampFn{Scalar min,max}`,
`ReductFn{size_t axis}`).

### 3b. DECIDED — binary vs broadcast: flag on the base (Option 1)

One subclass per op; the binary/expand axis is a **flag on
`ElementwiseBinaryFn`**, not a separate `Expand*` type.

```cpp
struct ElementwiseBinaryFn : LazyFunction {
  bool expand_;                                  // set in infer_output()
  bool is_fusible() const override { return !expand_; }   // v0: broadcast = barrier
};
struct AddFn : ElementwiseBinaryFn {
  void run_eager(TensorImpl&) override;          // → add_stub (re-routes internally)
  std::string codegen_expr() const override { return "{0}+{1}"; }
};
```

- **v0 fusible = same-shape elementwise only.** Barriers (captured, but
  realized eagerly via `run_eager`): matmul, reduct, broadcasting binaries,
  unary. Broadcast falls to the boundary path → existing `offset_util`.
- **Why:** binary/expand is orthogonal to the op and shared across all
  binary ops (both kernel handlers are functor-generic) — so it's a flag,
  not 22 types with routing duplicated into every shim.
- **Future-proof:** broadcast indexing is *data on the node*, not type
  structure. If it grows stateful, `bool expand_` → `IndexStrategy` is a
  local refactor (shims/types untouched). Roadmap fusions (matmul+epilogue,
  reduct+pro/epilogue) are orthogonal — partitioner/codegen + boundary
  node types, not this family.
- If the §0 "coarse static label" contract must stay literal: keep
  `is_fusible()` category-only and have the partitioner call a separate
  `can_fuse(node)` reading `expand_`. Same 11 classes; decide at impl time.

## 4. `record` (free fn) + concrete backend `realize` (realize stubbed)

> **Two steps, two modules** (see step list). **§4a `record`** is a
> `tensor` free fn and only needs §2 (the `lazy_` field) + §3
> (`infer_output`). **§4b `realize`** is the inductor backend and needs §0b
> (the lib to live in), §1 (the seam to subclass), and §3 (`run_eager`).
> §4a unblocks §5 (op shims); §4b unblocks §7 (registration). They can
> proceed independently once their own arrows are satisfied.

**record** (§4a — free fn in `tensor`, §0) — takes a `shared_ptr<LazyFunction>`,
calls its `infer_output()` (which computes output metadata *now* via
`meta_handler::infer()` — `type_upcast`, shape, broadcast flags — and
allocates a `TensorImpl` with a **0-byte** `Storage`), sets
`impl->lazy_ = fn`, and returns the lazy impl. No bytes, no backend.
Operands held as `shared_ptr<TensorImpl>` (`1.md §6`) so producers stay
alive.

`NVRTCInductorBackend : LazyBackend` (§4b — lives in `inductor`, §0b),
registered into `tensor` at startup (§7), provides **realize** only.

**realize(impl)** (§4b) — **Part 1b stub form:** DFS the operand graph; for each
operand that is itself lazy, realize it first — by re-entering the same
per-device chokepoint (`in->realize()` → `backend(in->device())->realize(in)`,
Decision 3), not a backend-private recursion. Then call the node's
`run_eager` bridge (→ existing `*_stub()`), write the resulting `Storage`
back into the impl, and clear `lazy_`. Because `.to` stays eager (Decision 4),
every operand of a lazy node shares that node's device (or is an
already-realized leaf), so this DFS never crosses a device boundary. This
reproduces the eager answer through the recorded graph with **no fusion**.
Part 2 replaces the body with the fusion-group partition + NVRTC codegen +
single-kernel launch, keeping `run_eager` as the boundary path.

Open Q (record in `index.md`): assume we never `data()` an *earlier*
node after it's been realized-and-cleared — hardcode that assumption,
assert if violated.

**Files.** `csrc/src/inductor/nvrtc/backend.cpp` (the `LazyBackend` subclass +
realize). `record` is a `tensor` free fn, not here. Registration fills the
per-device registry slot via `register_backend(DeviceType::CUDA, &inst)`
(Decision 1).

## 5. Op shim: the two-arm `defer` branch

Every op gateway (`ops::add`, `ops::neg`, …, and the shape ops from
`1.md §1`) gets the Option-A branch:

```cpp
Tensor ops::add(const Tensor& a, const Tensor& b) {
  using detail::UnsafeTensorAccessor;
  LMP_CHECK(a.device() == b.device()) << "...";              // validation: both paths
  if (backend(a.device()))                                   // backend for this device?
    return UnsafeTensorAccessor::fromImpl(record(std::make_shared<AddFn>(
        std::vector<std::shared_ptr<TensorImpl>>{
            UnsafeTensorAccessor::getImpl(a), UnsafeTensorAccessor::getImpl(b)})));
  return /* today's add_stub() body, verbatim */;
}
```

- The gate is `backend(a.device()) != nullptr` (Decision 2) — no `device ==
  CUDA` literal. No-backend-for-this-device cases take the eager arm →
  unchanged. The eager body below the guard is byte-for-byte today's code,
  which is what keeps the test suite green.
- Applies to **every op except cross-device `to`** (Decision 4) — boundary
  ops record too and realize via `run_eager`; `to` stays eager as the device
  barrier so no graph ever spans devices.
- Inputs come from `UnsafeTensorAccessor::getImpl(a)` (→
  `shared_ptr<TensorImpl>`, `1.md §6`); the result is wrapped with
  `::fromImpl`.
- **No macro/helper** (Decision 5): the guard is one line, written inline in
  each shim.

**Files.** `csrc/src/tensor/native/expand_ops.cpp` (binary),
`unary_ops.cpp`, `shape_ops.cpp`, `reduct_ops.cpp`, `matrix_ops.cpp`,
`conv_ops.cpp` — each `ops::*` definition (except `ops::to` in
`memory_ops.cpp`, Decision 4) gains the inline guard.

## 6. `data()` realize chokepoint

Fill the `1.md §5` stub: `data()` forces realization, then returns bytes.

```cpp
void* TensorImpl::data() {
  if (is_lazy()) backend(device())->realize(this);  // per-device realize (Decision 3)
  return data_.data();
}
```

- All host-side raw-byte access (`to_vector`, `.index()`, `print`,
  autograd reading saved tensors) bottoms out in `data()`, so this one
  chokepoint catches the whole host-read class.
- `realize()`/`sync()` public methods (from `1.md §5`) now delegate to
  `backend(device())->realize(this)`.
- Non-`data()` consumers (non-elementwise kernels that read operands
  directly, backward) still need their own triggers — flagged, handled as
  they surface / in Part 2's boundary realize.

**Files.** `csrc/src/tensor/tensor_impl.cpp`, `tensor.cpp`.

## 7. Static-registration / linker hazard — DEFERRED (out of scope this pass)

> **Decision 6 (2026-06-27): not scoped for this pass.** Use the existing
> static-registrar style (`LMP_REGISTER_DISPATCH`-shaped) for now and accept
> that cross-`.so` survival is unproven; robust handling is scoped
> separately later. The rest of this section is retained as context for that
> future work.
>
> Pairs with §0b: §0b stands up the `inductor_core` target, §7 makes sure
> its self-registering TU is not discarded across the `.so` boundary. The
> §0b "done when" (`backend(dev)` non-null at runtime) is exactly the check
> that §7 is working.

Already bitten once (commit `1cd2975` "linker error fix"). If inductor is
a separate shared lib and nothing in `tensor` references its symbols, the
linker discards the `register_backend()` static-init TU — `backend(dev)`
stays null and every op silently takes the eager arm.

Plan now (don't discover it at link time):
- Use `--whole-archive` / `-u` on the inductor lib, **or**
- Expose an explicit `lmp_inductor_init()` that `tensor` (or the app)
  calls, making the registrar transitively referenced.
- The cross-`.so` boundary weakens the usual static-registration macro
  trick (`dispatch_stub.hpp` comment) — prefer the explicit-init route.

**Files.** `CMakeLists.txt` (link flags / new inductor target), inductor
backend registration TU.

## Verification (end-to-end, no fusion yet)

1. No backend registered for a device → identical to today (eager
   everywhere on that device).
2. Backend registered for a device, tensors on it → ops return lazy impls
   (`is_lazy()==true`, 0-byte storage), and a forcing read (`to_vector`,
   `print`, `.index()`) yields **bit-identical results to the eager
   path**, proving record + `run_eager` realize is correct before any
   codegen exists. Because the gate is per-device (Decision 2), this is
   exercisable with a **CPU** lazy backend — no GPU required.
3. Tensors on a device with *no* registered backend → still eager (the gate
   is `backend(device) != nullptr`).
4. Mixed-device graph via `.to` → `.to` forces its input subgraph and emits
   a realized leaf on the target device (Decision 4); each single-device
   region is realized by exactly one backend, never two on one graph.

When this is green, Part 2 swaps the `realize` body from "walk + run_eager
per node" to "partition into fusion groups + one NVRTC kernel per group",
with `run_eager` retained as the boundary path.
