# Part 1 — LazyFunction / graph-shape design decisions

> Settles *what a lazy graph node is*, *where its metadata lives*, and
> *which module owns what*. Supersedes the `LazyNode` two-class sketch in
> `index.md` and `NOTES.md`. Feeds the mechanism in `2.md` (Part 1b).

## The lens: shadow the autograd graph exactly

The deferred-computation graph is a 1:1 mirror of the autograd graph we
already trust:

| Autograd                          | Lazy analogue                        |
|-----------------------------------|--------------------------------------|
| `VariableImpl {data, _grad_fn}`   | `TensorImpl {data_, lazy_}`          |
| `Function {saved_inputs}` + `apply` | `LazyFunction {inputs}` + `run_eager` |
| `ClampBackward {min_, max_}`      | per-op `LazyFunction` subclass        |

Autograd has **no separate node class** — `VariableImpl` *is* the vertex
and `Function` only holds the inputs + how to compute. We copy that.

## Decisions

### 1. No separate `LazyNode` — `TensorImpl` is the vertex
`is_lazy()` (`lazy_ != nullptr`) distinguishes the two flavors:
- **leaf**: realized, has `data_`, terminates DFS.
- **op node**: 0-byte `Storage`, real metadata, `lazy_` set.

The old leaf/op-node split collapses into `TensorImpl`. DFS terminates on
`!is_lazy()` instead of a leaf-wrapper object.

### 2. Metadata lives in `TensorImpl`, never duplicated in `LazyFunction`
`shape_`, `type_`, `strides_`, `numel_` already sit on `TensorImpl`;
device comes from `Storage` (no `device_` field — same as today). The
output impl from `record()` carries all of it (0-byte storage, real
shape/dtype/device), and operands are `TensorImpl`s carrying their own —
so realize/codegen reach every node's metadata without `LazyFunction`
storing any. (Autograd's `Function` likewise carries zero output
metadata.) The user can call `lazy_tensor.shape()` pre-realize and it must
*not* force — another reason the metadata has to be on the impl.

### 3. No `OpKind` enum
Op behavior (`run_eager`, `codegen_expr`) is carried by the
`LazyFunction` subclass via vtable — exactly like autograd's `Function`
subclasses. There is **no execution dispatch on a kind tag** (that's the
anti-pattern: every new field/op touches every switch arm). The only tag
is a coarse, stable `is_fusible()` / `OpCategory` label (≈5 values, not
one-per-op) for Part 2's partitioner to ask "fusible or boundary?"
without RTTI.

### 4. `LazyFunction` + all op subclasses live in `tensor` (not inductor)
Forced by direct construction: `record(make_shared<AddFn>(...))` can only
compile where `AddFn` is visible, and the op-shim `ops::add` is public
`tensor` API that must stay in `tensor`. The two ways to keep subclasses
in `inductor` are both rejected:
- a type-erased key so the shim names the op without seeing it → that was
  `OpKind` (deleted), and
- a per-op virtual on the backend (`backend->add`, `->sub`, … ×30) → the
  cross-module surface blow-up we're explicitly avoiding.

**Vocabulary vs machinery split** (why this isn't "fusion logic in
tensor"):
- `tensor` holds op **vocabulary** — per-op, grows as ops are added:
  `inputs`, `run_eager` (→ existing `*_stub()`), `codegen_expr` string,
  `is_fusible` label. Holding this doesn't teach tensor to fuse; it lets
  ops describe themselves. **Adding an op = one file, one module.**
- `inductor` holds the fusion **machinery** — fixed complexity, never
  grows per op: graph partition / group merge, NVRTC codegen, module
  cache, kernel launch.

Cross-module surface = **exactly one seam: `RealizationBackend`.**
`inductor` includes `tensor`, reads `LazyFunction`, and exposes nothing
back. `tensor` never includes `inductor`.

### 5. `record` returns `shared_ptr<TensorImpl>`, is a free fn, needs no backend
Each subclass owns its shape logic via `infer_output()` (builds the
0-byte impl through `meta_handler::infer()` from `1.md §2`). So `record`
is thin: call `fn->infer_output()`, set `impl->lazy_ = fn`, return the
impl. Shape inference is shared on a category base
(`ElementwiseBinaryFn`), not rewritten per op.

### 6. Singleton `RealizationBackend* backend()` stays — for realize, not record
Needed for `realize()` (NVRTC) and as the **defer gate**
(`backend() && device == CUDA`); construction goes through the free
`record`, no virtual. Static-registration/linker hazard (commit
`1cd2975`) still applies → explicit-init route, see `2.md §7`.

## Sketch

```cpp
// tensor module
struct LazyFunction {                          // abstract
  std::vector<std::shared_ptr<TensorImpl>> inputs;
  virtual ~LazyFunction() = default;
  virtual std::shared_ptr<TensorImpl> infer_output() const = 0; // 0-byte impl
  virtual void run_eager(TensorImpl& out) = 0;                  // → *_stub()
  virtual std::string codegen_expr() const = 0;                 // Part 2 seam
  virtual bool is_fusible() const = 0;                          // coarse label
};

struct ElementwiseBinaryFn : LazyFunction {    // broadcast infer once, here
  std::shared_ptr<TensorImpl> infer_output() const override;
  bool is_fusible() const override { return true; }
};

struct AddFn : ElementwiseBinaryFn {           // the whole op, one place
  void run_eager(TensorImpl& out) override;    // add_stub on realized inputs
  std::string codegen_expr() const override { return "{0}+{1}"; }
};

// record: free fn in tensor, no backend needed
std::shared_ptr<TensorImpl> record(std::shared_ptr<LazyFunction> fn) {
  auto out = fn->infer_output();
  out->lazy_ = std::move(fn);
  return out;
}

Tensor ops::add(const Tensor& a, const Tensor& b) {
  bool defer = backend() && a.device() == DeviceType::CUDA;
  if (!defer) return primitives::add(a, b);    // today's eager path
  return Tensor{ record(std::make_shared<AddFn>(
      std::vector{a.impl(), b.impl()})) };
}

// realize stays behind the seam (inductor)
void* TensorImpl::data() {
  if (is_lazy()) backend()->realize(this);     // force, then fall through
  return data_.data();
}
```

## Pointers vs values

| Thing                         | Type                                   | Why |
|-------------------------------|----------------------------------------|-----|
| `TensorImpl::lazy_`           | `shared_ptr<LazyFunction>`             | DAG, not tree (≈ `_grad_fn`) |
| `LazyFunction::inputs`        | `vector<shared_ptr<TensorImpl>>`       | keep producers alive to realize |
| shape / dtype / strides       | by value on `TensorImpl`               | already are |
| op params (clamp bounds, axis)| by value on the subclass               | ≈ `ClampBackward::min_val_` |

Graph edges point output→inputs only (acyclic) → plain `shared_ptr`
everywhere, no `weak_ptr` needed for the lazy graph.
