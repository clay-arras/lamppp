# Part 1a — Library refactor (no lazy semantics yet)

> Prerequisite refactors that make the existing eager library *capable* of
> deferral, without introducing any laziness. After this part the library
> still behaves identically — every op still evaluates eagerly — but the
> seams the inductor needs (`2.md`) exist.
>
> These items describe *what* changes and *why*; the design/mechanism is
> deliberately left open. Verification target for the whole part: **all
> existing tensor/autograd tests pass unchanged** — nothing here changes
> observable behavior.

---

## 1. Shape/view ops route through `ops::`

**What.** `reshape`/`squeeze`/`expand_dims` should go through the
`ops::` layer like every other op.

**Why.** Today `Tensor::reshape/squeeze/expand_dims`
(`csrc/src/tensor/tensor.cpp:25-36`) call `impl_->reshape(...)` directly,
bypassing the already-existing `ops::reshape/squeeze/expand_dims`
(`csrc/src/tensor/native/shape_ops.cpp`). Because they don't pass through
`ops::`, a capture layer can't see them.

**Done when.** No view is constructed by calling `impl_->reshape` (etc.)
outside the `ops::` layer; tests unchanged.

---

## 2. `meta_handler` can produce metadata without allocating

**What.** It must be possible to obtain an op's output metadata
(dtype/shape/size and broadcast info) without allocating its output
`Storage`.

**Why.** Each `TensorMetaHandler` ctor
(`csrc/src/tensor/cpu/meta_handler.cpp`) currently computes metadata *and*
allocates storage in one step. The capture path needs the metadata alone
(no bytes), while the eager path still needs the allocated output.

**Done when.** Metadata can be computed for an op without its output
buffer being allocated; the eager kernels still get a fully allocated
output.

---

## 3. `Storage` supports a zero-size allocation on a device

**What.** A `Storage` can record a device while holding zero bytes.

**Why.** A deferred tensor needs a `Storage` that knows its device but has
no data yet. Today `Storage`/`StorageImpl`
(`csrc/include/lamppp/tensor/storage.hpp`) always allocates `byte_size`
bytes via the device allocator.

**Done when.** A zero-byte `Storage` on a device constructs and reports
its device correctly, and can later be grown to real bytes.

---

## 4. `TensorImpl` tolerates a zero-byte storage

**What.** The `TensorImpl` storage-size invariant must permit a zero-byte
`Storage` paired with a non-zero shape.

**Why.** `TensorImpl(Storage, shape, dtype)`
(`csrc/src/tensor/tensor_impl.cpp:11-28`) asserts that storage size
matches `numel`. A deferred impl carries a zero-byte storage but a real
shape, which trips that assertion.

**Done when.** A `TensorImpl` with a real shape and a zero-byte storage
constructs successfully, while a genuinely wrong-sized storage still
fails.

---

## 5. `data()` must be able to realize

**What.** `data()` can no longer be a pure read — it must be allowed to
mutate the impl (it will become the point where a deferred tensor is
forced).

**Why.** `TensorImpl::data()` and `Tensor::data()` are `const noexcept`
(`tensor_impl.hpp:72`, `tensor.hpp:46`). A const/noexcept accessor can't
trigger realization or allocate.

**Done when.** `data()` is no longer `const`/`noexcept` and the project
compiles with that change; behavior is unchanged (no realization logic
yet).

---

## Sequencing

Items are largely independent. Each must keep the full test suite green on
its own — this part introduces **zero** behavioral change. The lazy
machinery (`2.md`) builds on these seams.
