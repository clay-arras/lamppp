# Part 1a · Item 2 — `meta_handler` infers metadata without allocating

> Mechanism for `1.md §2`. The dependency list in `2.md` names this
> `meta_handler::infer()`. This file fleshes out the *how*; `1.md` left it
> deliberately open. Verification target is unchanged: **all existing
> tensor/autograd tests pass unchanged** — zero behavioral change.

## What's entangled today

`TensorMetaHandler<Args...>` has three specializations
(`UnaryMetaHandler` / `BinaryMetaHandler` / `ReductMetaHandler`) in
`csrc/src/tensor/cpu/meta_handler.cpp`. Each ctor does two distinct jobs:

1. **Infer** (cheap, pure) — `outDtype_`, `outSize_`, `outShape_`,
   `expand_`. Already computed in the initializer list / pre-dispatch
   body.
2. **Allocate** (the part the capture path must avoid) — the
   `LMP_DISPATCH_ALL_TYPES { Storage out_st(...); outTen_ =
   make_unique<TensorImpl>(...) }` block, plus the binary-expand
   `outOffset_` computation.

They're *almost* already separated: the dispatch block is purely
allocation. The only reason allocation needs a type-dispatch is
`sizeof(out_dtype_t)` for byte size and building `outTen_`. The offset
table (`offset_util_stub_2`) is the one wrinkle — it currently needs an
output `TensorImpl`.

Notes that bound the blast radius:

- `meta_handler.cpp` is **shared by CPU and CUDA** — the `cuda/*.cu`
  dispatch handlers take `BinaryMetaHandler` etc. too.
- matmul / conv / transpose build their outputs by hand and **don't** use
  the handler, so this item only touches the elementwise/reduct path.

## Design (Option B — the `infer()` seam)

Extract a pure metadata struct + `infer_*` helpers; leave the allocating
ctor behavior identical.

```cpp
// meta_handler.hpp
struct OpMeta {
  DataType dtype;
  size_t size;
  std::vector<size_t> shape;
  bool expand;
};

OpMeta infer_unary (const TensorImpl* a);
OpMeta infer_binary(const TensorImpl* a, const TensorImpl* b);
OpMeta infer_reduct(const TensorImpl* a, size_t axis);
```

Each existing ctor is refactored to *consume* its `infer_*` result, then
allocate exactly as before:

```cpp
BinaryMetaHandler::TensorMetaHandler(const TensorImpl* a, const TensorImpl* b)
    : inTens_({a, b}) {
  OpMeta m = infer_binary(a, b);
  outDtype_ = m.dtype; outSize_ = m.size; outShape_ = m.shape; expand_ = m.expand;
  LMP_DISPATCH_ALL_TYPES(...) { /* Storage + outTen_ + offset, unchanged */ }
}
```

- **Eager path: zero change.** `kernels.cpp` / `kernels.cu` still do
  `TensorMetaHandler meta(&a, &b)` → fully allocated `out()`. The hot
  path, the `noexcept`-ness of `out()`, and all ~50 explicit
  instantiations stay untouched. This is what keeps "tests pass
  unchanged" cheap and low-risk.
- **Capture path (`2.md §4` `record`): calls `infer_binary(a, b)` only** —
  gets dtype/shape/size/broadcast-flag, no `Storage`, no offset table.

### Why not Option A (lazy-allocating `out()`)

Making the ctor infer-only and having `out()` lazily allocate on first
call would force `out()` non-`const` / non-`noexcept` and hide allocation
in an accessor the eager kernels call repeatedly — more churn on the hot
path for no benefit, since the capture path is a *different* call site
anyway. Option B is a cleaner seam and is exactly the `infer()` named in
`2.md`'s dependency list.

## Offset-table handling (explicit decision)

`outOffset_` (broadcast index map via `offset_util_stub_2`) is **not**
part of `OpMeta`. The record path needs only the `expand` flag (`2.md §4`:
"broadcast flags"); the actual offset table is codegen/realize-time
(Part 2), and Part 1b's `run_eager` bridge recomputes its own meta anyway.
So `infer_*` deliberately stops at the flag — this is a decision, not an
omission.
